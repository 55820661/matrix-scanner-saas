# خطة تصحيح المسار - Matrix Scanner SaaS

من النقطة الحالية إلى إعداد Telegram Bot

الإصدار: 1.0  
تاريخ الوثيقة: 4 يونيو 2026  
المصدر: `docs/planning/خطة_تصحيح_المسار_Matrix_Scanner_SaaS.docx`

الغرض من هذه الوثيقة هو تثبيت المسار التنفيذي الصحيح للمشروع بعد تصحيح المفهوم: يوجد AI واحد فقط داخل شات الأدمن، بينما باقي الأجزاء هي طبقات تشغيل وتحكم وتخزين وتكامل. كما تثبت الوثيقة تصحيحا إضافيا لمفهوم الأدوات: الأداة في المرحلة الحالية ليست بالضرورة runtime handler مستقل داخل كود الـ Agent، بل هي في الأساس command template أو script template read-only محفوظ ومنظم داخل Tool Registry، وينفذه Runtime/Agent بأمان بعد تحقق SaaS من السياسات.

## 1. الملخص التنفيذي

هذه الوثيقة تحدد الخطة التنفيذية الكاملة لمشروع Matrix Scanner SaaS من الوضع الحالي حتى إعداد Telegram Bot، مع الالتزام بعدم هدم ما تم إنجازه سابقا، بل استخدامه كأساس للتطوير التالي.

الفكرة المحورية بعد التصحيح:

```text
لا توجد عدة عقول ذكاء اصطناعي داخل المشروع.
يوجد عقل واحد فقط هو Admin AI Chatbot داخل الأدمن/البورتال.
```

هذا العقل الواحد مسؤول عن:

- المناقشة.
- التشخيص.
- اختيار الأدوات.
- طلب تشغيل الأدوات.
- اقتراح أدوات جديدة.
- اختبار وتنقيح الأدوات.
- توليد التقارير.

أما Tool Registry وTool Runtime وReports وTelegram فهي مكونات خدمية حول هذا العقل الواحد، وليست AI مستقلة.

## 1.1 تصحيح مفهوم الأدوات والـ Runtime

يجب ألا تفترض الخطة أن كل أداة جديدة تحتاج إلى runtime handler مستقل داخل كود الـ Agent. التصور المعتمد للمرحلة الحالية هو:

```text
Tool = Approved read-only command template
```

وليس:

```text
Tool = Runtime handler code فقط
```

الأداة داخل Tool Registry يمكن أن تحتوي:

- `tool_key`
- اسم ووصف الأداة.
- نوع السيرفر أو البروفايل المناسب.
- الأمر أو السكربت read-only.
- المدخلات المسموحة وقواعد التحقق منها.
- timeout.
- max output size.
- redaction rules.
- expected output format.
- requires approval.
- plan availability.
- risk level.
- أمثلة آمنة للاستخدام.

التدفق الصحيح:

```text
Admin AI Chatbot يختار أو يقترح أداة
SaaS يتحقق من ToolDefinition/ToolPolicy/PlanTool والصلاحيات
Runtime/Agent ينفذ command template المعتمد بأمان
النتيجة تعود إلى SaaS ثم إلى Admin AI Chatbot للتحليل
```

Runtime handlers تظل خيارا متقدما لاحقا للحالات التي تحتاج parser أو collector معقد داخل الكود، لكنها ليست الأساس الحالي.

### الشكل المقترح للأداة

الأداة داخل Tool Registry يجب أن تكون وصفا منظما وقابلا للتحقق، وليس أمرا حرا. الشكل المتوقع يتضمن:

- `tool_key`
- اسم الأداة.
- وصف الأداة.
- نوع السيرفر أو البروفايل المناسب.
- الأمر أو السكربت read-only.
- المدخلات المسموحة إن وجدت.
- timeout.
- max output size.
- redaction rules.
- expected output format.
- هل تحتاج approval قبل التشغيل.
- هل هي مفعلة لخطة معينة.
- مستوى الخطورة.
- أمثلة آمنة للاستخدام.

مثال:

```text
tool_key: nginx_error_tail
description: Read recent Nginx error log lines safely
command: tail -n {{lines}} /var/log/nginx/error.log
inputs:
  lines:
    type: integer
    min: 20
    max: 300
read_only: true
timeout_seconds: 10
max_output_chars: 20000
requires_approval: false
```

هذا المثال لا يعني السماح بأوامر حرة. الأمر محفوظ كـ template، والمدخلات محدودة، والتحقق يتم قبل التنفيذ.

### دور Admin AI Chatbot في الأدوات

Admin AI Chatbot هو العقل الوحيد داخل النظام. دوره في مسار الأدوات:

1. يفهم المشكلة من كلام الأدمن.
2. يراجع Safe Context.
3. يبحث عن أداة مناسبة داخل Tool Registry.
4. يختار الأداة المناسبة أو يقترح أداة جديدة.
5. إذا لم توجد أداة مناسبة، ينشئ command template جديد كـ draft/proposal.
6. يحدد لماذا يحتاج هذه الأداة.
7. يحدد هل الأمر read-only أم لا.
8. يحدد المدخلات والمخرجات المتوقعة.
9. يعرض الأداة المقترحة للاعتماد حسب السياسة.
10. بعد الاعتماد، يطلب تشغيلها من خلال ToolRun/AgentJob.
11. يستقبل النتيجة.
12. يحلل الناتج ويقرر الخطوة التالية.
13. ينشئ تقريرا تقنيا أو مبسطا عند الطلب.

### دور Runtime / Agent

الوصف المعتمد:

```text
Runtime/Agent = Safe restricted command executor
```

وليس بالضرورة مكتبة handlers لكل أداة.

وظائفه:

- يستقبل AgentJob من SaaS.
- يحصل على الأمر المعتمد أو command template بعد تحقق SaaS من السياسات.
- ينفذ الأمر على السيرفر بصلاحيات محددة.
- يمنع الأوامر غير المصرح بها.
- يطبق timeout.
- يحد من حجم المخرجات.
- يرجع stdout/stderr/exit_code/execution_time بشكل منظم وآمن.
- لا يخترع أوامر من نفسه.
- لا ينفذ shell حر.
- لا ينفذ destructive commands في هذه المرحلة.

### علاقة Tool Registry بالـ Runtime

العلاقة المعتمدة:

1. Tool Registry في SaaS هو مصدر الحقيقة للأدوات.
2. Runtime لا يشغل إلا أدوات معتمدة ومرسلة من SaaS ضمن AgentJob.
3. SaaS يتحقق قبل التشغيل من:
   - ToolDefinition.
   - ToolPolicy.
   - PlanTool.
   - صلاحية المستخدم.
   - صلاحية السيرفر.
   - كون الأداة read-only.
4. Runtime ينفذ الأمر كما هو بعد تطبيق القيود.
5. النتيجة ترجع إلى SaaS ثم إلى Admin AI Chatbot.

### إنشاء أدوات جديدة

عندما يحتاج Admin AI Chatbot أداة جديدة، لا يتم تشغيل أمر جديد فورا على السيرفر.

المسار الصحيح:

1. Admin AI Chatbot يقترح command template.
2. يتم حفظه كـ ToolBuildRequest أو ToolBuildProposal.
3. يتم إجراء validation على الأمر:
   - read-only.
   - لا يحتوي حذف.
   - لا يحتوي تعديل.
   - لا يحتوي restart.
   - لا يحتوي secrets access.
   - لا يحتوي network exfiltration.
   - لا يحتوي pipes خطرة غير مبررة.
4. يتم اختباره في بيئة آمنة أو على نطاق محدود حسب السياسة.
5. بعد الاعتماد، يتحول إلى ToolDefinition مفعلة.
6. بعدها فقط يمكن تشغيله من خلال ToolRun/AgentJob.

### تصنيف الأدوات

تصنيف الأدوات في هذه المرحلة:

1. `command_template` - الأساس الحالي.
2. `script_template` - مجموعة أوامر قصيرة منظمة.
3. `runtime_handler` - أدوات متقدمة لاحقا عند الحاجة.

الصياغة المعتمدة:

```text
الأدوات في المرحلة الحالية هي أوامر أو سكربتات read-only محفوظة كـ Command Templates داخل Tool Registry.
يقوم Admin AI Chatbot باختيارها أو اقتراحها.
يقوم SaaS بالتحقق من السياسات.
يقوم Runtime/Agent بتنفيذها بأمان وإرجاع النتائج.
أما Runtime handlers فهي خيار متقدم لاحق وليست الأساس الحالي.
```

## 2. القاعدة الحاكمة للخطة

- لا نهدم أي جزء من SaaS Core أو Agent أو Tool Registry أو Baseline أو Portal أو Reports.
- لا ننشئ AI مستقل للتشخيص أو التقارير أو بناء الأدوات.
- نستخدم Diagnostic Agent الحالي كبنية جلسات وقرارات، وليس كعقل مستقل.
- نستخدم Tool Builder الحالي كمسار منظم داخل نفس شات الأدمن.
- نستخدم Telegram لاحقا كواجهة خارجية لنفس Admin AI Chatbot.
- كل أدوات التنفيذ تظل read-only في هذه المرحلة.
- الأدوات الحالية والمفضلة في هذه المرحلة هي command templates أو script templates منظمة داخل Tool Registry.
- Runtime handlers خيار متقدم لاحق فقط عند الحاجة.
- لا توجد أوامر shell حرة أو destructive actions.

## 3. الوضع الحالي الذي سيتم البناء عليه

- SaaS Core موجود ويعمل.
- Admin وPortal موجودان وتم اختبارهما.
- Agent Runtime موجود ويستطيع polling وتنفيذ AgentJob.
- Tool Registry وToolPolicy وPlanTool موجودة.
- Baseline Scan موجود، والبروفايل `debian_nginx_opt` يعمل end-to-end.
- النتائج تخزن في نماذج منظمة مثل services وdomains وapplications وlog sources.
- Reports foundation موجودة ويمكن استخدامها لاحقا لتقارير تقنية ومبسطة.
- Telegram foundation موجودة جزئيا، وسيتم تطويرها لاحقا كواجهة لنفس الشات بوت.

## 4. المعمارية المصححة

المشروع بعد التصحيح يتكون من AI واحد وخمس طبقات خدمية حوله:

| المكون | الدور | المسؤولية |
|---|---|---|
| Admin AI Chatbot | العقل الوحيد داخل النظام | المناقشة، التشخيص، اختيار الأدوات، طلب تشغيلها، اقتراح أدوات جديدة، تبسيط النتائج، إنشاء التقارير. |
| SaaS Backend | طبقة الإدارة والتحكم | إدارة الحسابات والسيرفرات والصلاحيات والسياسات والتخزين وسير العمل. |
| Tool Registry / Policy | مخزن الأدوات وقواعد السماح | تعريف الأدوات، ربطها بالخطط، تحديد صلاحيات التشغيل، منع الأدوات غير المسموحة. |
| Internal Runtime / Agent | ذراع التنفيذ الآمن على السيرفر | تنفيذ command templates أو script templates المعتمدة read-only، تطبيق timeout/output caps، إرجاع stdout/stderr/exit_code/execution_time بشكل آمن، والعمل وفق أوامر المنصة فقط بدون تفكير مستقل. |
| Reports Engine | إنتاج التقارير | توليد تقارير تقنية وتبسيطها للعميل من خلال نفس الشات بوت. |
| Telegram Bot | واجهة خارجية لاحقة | استخدام نفس Admin AI Chatbot من خارج لوحة التحكم مع نفس الصلاحيات والسياسات. |

## 6. المرحلة 0 - تثبيت الوضع الحالي

الهدف: تثبيت ما تم إنجازه قبل أي تعديل حتى لا يتم كسر أساس المشروع أو تكرار أعمال انتهت بالفعل.

المهام التنفيذية:

- تأكيد آخر commit محلي وعلى السيرفر.
- تأكيد تشغيل الخدمات الأساسية.
- اختبار Admin وPortal.
- اختبار Agent polling.
- اختبار Baseline scan.
- اختبار ToolRun إلى AgentJob.
- توثيق الأجزاء الجاهزة والأجزاء المؤجلة.

المخرجات:

- تقرير قصير بحالة المشروع الحالية.
- قائمة مكونات لا يجب المساس بها إلا عند الحاجة.
- نقطة انطلاق ثابتة للتطوير التالي.

معيار النجاح:

- كل الخدمات الأساسية active.
- Baseline scan يعمل.
- لا توجد migrations غير متوقعة.
- Admin وPortal يفتحان بدون أخطاء.

## 7. المرحلة 1 - إعادة تعريف المعمارية

الهدف: تثبيت المفهوم الصحيح داخل وثائق المشروع: يوجد Admin AI Chatbot واحد فقط، والباقي مكونات تشغيلية وليست عقول مستقلة.

المهام التنفيذية:

- إنشاء أو تحديث وثيقة `docs/planning/ROADMAP-CORRECTION.md`.
- توضيح دور كل مكون في المعمارية.
- تحديد أن Telegram لاحقا واجهة لنفس الـ AI.
- تحديث أي مصطلحات داخل التوثيق تسبب لبسا مثل AI Diagnostic مستقل أو AI Report مستقل.

المخرجات:

- وثيقة معمارية مختصرة ومعتمدة.
- إزالة الالتباس من الخطة القادمة.
- توحيد لغة الفريق عند تنفيذ المراحل التالية.

معيار النجاح:

- لا تظهر الخطة وكأن المشروع يحتوي على ثلاثة AI.
- كل وظائف التشخيص والأدوات والتقارير تندرج تحت Admin AI Chatbot.

## 8. المرحلة 2 - تجهيز Safe Context Builder

الهدف: تجهيز سياق آمن ومنظم يقرأه Admin AI Chatbot بدل الوصول إلى raw logs أو أسرار أو مخرجات خام.

المهام التنفيذية:

- بناء طبقة تجمع ملخصات السيرفر.
- تلخيص التطبيقات والدومينات والخدمات ومصادر اللوجات.
- تلخيص findings وreports وآخر baseline.
- إضافة قائمة available tools وrecent runs.
- منع raw logs وraw `.env` وcredentials وtokens.

المخرجات:

- `server_summary`
- `applications_summary`
- `services_summary`
- `domains_summary`
- `log_sources_summary`
- `findings_summary`
- `risk_summary`
- `available_tools`
- `recent_runs`

معيار النجاح:

- السياق يكفي للـ AI لفهم الحالة.
- لا يحتوي السياق على أسرار.
- السياق محدود الحجم ومنظم وقابل للاختبار.

## 9. المرحلة 3 - بناء شات الأدمن الداخلي

الهدف: توفير واجهة محادثة داخل الأدمن أو البورتال يتعامل معها المستخدم مع Admin AI Chatbot مباشرة.

المهام التنفيذية:

- إنشاء Conversation Session.
- تخزين Message History.
- ربط الجلسة بالمستخدم والحساب والصلاحيات.
- السماح باختيار Server/Application.
- تحميل Safe Context حسب السيرفر المختار.
- تحديد صلاحيات Owner/Operator/Viewer.

المخرجات:

- صفحة شات داخلية.
- جلسات محادثة محفوظة.
- محادثة مربوطة بسيرفر أو تطبيق محدد.
- أساس عملي لكل المراحل التالية.

معيار النجاح:

- الأدمن يستطيع سؤال الشات عن حالة السيرفر.
- الشات يرد بناء على Safe Context فقط.
- لا يتم تشغيل أدوات في هذه المرحلة إلا إذا تم ربط Orchestrator لاحقا.

## 10. المرحلة 4 - AI Tool Orchestrator

الهدف: تمكين Admin AI Chatbot من استخدام الأدوات الموجودة حسب نوع المهمة مع الالتزام بالسياسات والصلاحيات.

المهام التنفيذية:

- تحليل طلب المستخدم لتحديد الحاجة إلى أداة.
- مطابقة الطلب مع available tools.
- عرض سبب اختيار الأداة.
- طلب موافقة عند الحاجة.
- التحقق من ToolPolicy وPlanTool وصلاحية المستخدم.
- إنشاء ToolRun ثم AgentJob.
- استقبال النتيجة وتفسيرها للمستخدم.

المخرجات:

- مسار تشغيل أداة من داخل الشات.
- نتيجة أداة تعود للـ AI.
- شرح بشري للنتيجة بدل raw output.

معيار النجاح:

- لا يتم تنفيذ shell حر.
- لا تعمل أداة غير مفعلة في الخطة.
- كل تشغيل موثق في ToolRun/AuditLog.

## 11. المرحلة 5 - Safe Command Execution Runtime

الهدف: جعل Runtime/Agent على السيرفر command executor آمنا ومقيدا للأدوات التي يطلبها Admin AI Chatbot بعد اعتمادها في SaaS.

المهام التنفيذية:

- دعم تنفيذ command templates المعتمدة فقط.
- منع shell الحر والأوامر غير المصرح بها.
- دعم timeout وmax output size.
- دعم parameter validation قبل التنفيذ.
- دعم redaction قبل التخزين أو العرض.
- إرجاع `stdout`, `stderr`, `exit_code`, و`execution_time` بصيغة منظمة وآمنة.
- تسجيل audit مناسب في SaaS حول تشغيل الأداة ونتيجتها.
- إبقاء runtime handlers كخيار متقدم لاحق للأدوات التي تحتاج parser أو collector معقد داخل الكود.

المخرجات:

- Runtime أكثر وضوحا واستقرارا كمنفذ أوامر آمن.
- ربط واضح بين ToolDefinition والسياسة وcommand template والتنفيذ.
- تقليل الحاجة إلى كتابة handler جديد لكل أداة بسيطة.

معيار النجاح:

- كل command template لا يعمل إلا بعد اعتماد ToolPolicy وPlanTool.
- كل نتيجة تعود بتنسيق مفهوم وآمن.
- لا توجد أدوات تعمل خارج السياسة.

## 12. المرحلة 6 - تمكين الشات بوت من مناقشة وإنشاء الأدوات

الهدف: إدخال Tool Builder الحالي داخل تدفق الشات بحيث يناقش AI الحاجة إلى الأداة ثم يحولها إلى طلب ومقترح منظم.

المهام التنفيذية:

- مناقشة الحاجة إلى أداة جديدة.
- فحص هل توجد أداة حالية تكفي.
- تحديد نطاق الأداة وكونها read-only.
- اقتراح command template أو script template منظم، وليس runtime handler افتراضيا.
- تحديد المخرجات الآمنة.
- إنشاء ToolBuildRequest.
- إنشاء ToolBuildProposal.
- تسليم التنفيذ إلى Codex أو مطور.
- إضافة اختبارات ومراجعة أمان قبل التفعيل.

المخرجات:

- ToolBuildRequest من داخل الشات.
- ToolBuildProposal منظم.
- مسار مراجعة بشرية.
- تفعيل Pilot فقط بعد النجاح.

معيار النجاح:

- لا يتم إنشاء أداة عشوائية بدون مراجعة.
- لا يتم تفعيل أداة بدون command template معتمد واختبار أمان. Runtime handler لا يكون مطلوبا إلا للأدوات المتقدمة.
- كل أداة جديدة تظل read-only في هذه المرحلة.

## 13. المرحلة 7 - أول دورة أداة كاملة

الهدف: اختبار المسار كاملا على أداة واحدة أو أداتين فقط لإثبات أن الشات قادر على فهم الاحتياج وإدارة دورة الأداة حتى التشغيل.

المهام التنفيذية:

- اختيار أداة مناسبة كبداية.
- مناقشتها داخل الشات.
- إنشاء ToolBuildRequest وToolBuildProposal.
- تنفيذها بواسطة Codex أو مطور.
- اختبارها محليا وعلى السيرفر.
- مراجعة الأمان.
- تفعيلها للـ pilot.
- تشغيلها من الشات وشرح نتيجتها.

المخرجات:

- أداة واحدة تعمل end-to-end.
- توثيق دورة الأداة كاملة.
- تثبيت شكل التنفيذ اللاحق لباقي الأدوات.

معيار النجاح:

- الأداة تعمل من الشات.
- النتيجة تعود للـ AI.
- الـ AI يشرح النتيجة بلغة بشرية.
- لا توجد بيانات حساسة في المخرجات.

أدوات مناسبة لأول دورة:

- Laravel Log Health.
- Apache 5xx Summary.
- Supervisor Summary.
- Queue Workers Summary.
- Laravel Env Sanity.

السيرفر المقترح لهذه المرحلة: سيرفر إنفي أو سيرفر Laravel/Apache/cPanel مشابه، لأنه يمثل نوعا مهما من بيئات العملاء التجاريين.

## 14. المرحلة 8 - التقارير من داخل نفس الشات بوت

الهدف: تحويل التقارير إلى قدرة داخل Admin AI Chatbot، وليست AI مستقل، بحيث يستطيع إنتاج تقرير تقني مفصل أو تقرير مبسط للعميل.

المهام التنفيذية:

- استخدام Reports foundation الموجود.
- إضافة report templates.
- تمكين AI من بناء sections.
- إتاحة مراجعة التقرير قبل الحفظ.
- حفظ التقرير داخل النظام.
- توليد نسختين: تقنية ومبسطة.

المخرجات:

- تقرير تقني مفصل.
- تقرير مبسط للعميل.
- أقسام محفوظة ومراجعة.
- صياغة مفهومة وغير حساسة.

معيار النجاح:

- التقرير لا يحتوي raw logs أو أسرار.
- التقرير الفني مفيد للمطورين.
- التقرير المبسط مفهوم للعميل غير التقني.

## 15. المرحلة 9 - Pilot داخلي كامل

الهدف: تشغيل النظام داخليا على أكثر من بيئة حقيقية قبل فتح Telegram.

المهام التنفيذية:

- اختبار Debian/Nginx/Opt على سيرفر Matrix/Siyaaq.
- اختبار Laravel/Apache/cPanel على إنفي أو بديل.
- اختبار Laravel/Nginx عند الحاجة.
- قياس جودة اختيار الأدوات.
- قياس جودة التقارير.
- قياس أمان المخرجات.
- تقييم استقرار runtime.

المخرجات:

- تقرير Pilot داخلي.
- قائمة تحسينات قبل Telegram.
- قرار جاهزية Telegram.

معيار النجاح:

- الشات يفهم السيرفرات الحقيقية.
- الأدوات تعمل بثبات.
- النتائج آمنة.
- التقارير مفهومة.

## 16. المرحلة 10 - تجهيز Telegram كواجهة خارجية لنفس الـ AI

الهدف: فتح نفس Admin AI Chatbot من خلال Telegram Bot بدون بناء منطق مستقل أو AI منفصل.

المهام التنفيذية:

- ربط Telegram chat بالمستخدم والحساب.
- تطبيق نفس الصلاحيات.
- اختيار السيرفر من Telegram.
- إرسال الرسائل لنفس AI Orchestrator.
- استخدام نفس Safe Context.
- استخدام نفس ToolPolicy وToolRun وAgentJob.
- إرجاع النتائج بصيغة مناسبة لتليجرام.

المخرجات:

- Telegram Bot متصل بنفس الشات بوت.
- أوامر أولية آمنة.
- تشغيل محدود في private chat.

معيار النجاح:

- لا group diagnostics في البداية.
- لا raw logs.
- لا raw env.
- لا تشغيل أدوات حساسة بدون approval.
- نفس قواعد الأدمن تطبق على Telegram.

أوامر Telegram المقترحة للبداية:

- `/start`
- `/help`
- `/servers`
- `/select_server`
- `/status`
- `/diagnose`
- `/report`
- `/cancel`

## 17. المرحلة 11 - Telegram Pilot

الهدف: اختبار Telegram مع مستخدم داخلي فقط قبل أي استخدام أوسع.

المهام التنفيذية:

- السؤال عن حالة السيرفر.
- طلب تشخيص مشكلة.
- طلب تشغيل أداة read-only.
- طلب تقرير مبسط.
- طلب تقرير تقني.
- اختبار عدم وجود صلاحية.
- اختبار طلب خطير والتأكد من رفضه.
- اختبار طلب raw logs والتأكد من عدم عرضها.

المخرجات:

- نتيجة اختبار Telegram Pilot.
- قائمة مشاكل واجهة Telegram.
- قرار الانتقال لاستخدام أوسع أو استمرار التحسين.

معيار النجاح:

- Telegram يستخدم نفس AI ونفس السياسات.
- كل الردود آمنة ومفهومة.
- لا توجد قناة التفاف على صلاحيات الأدمن.

## 18. القيود الأمنية الثابتة

- لا raw `.env`.
- لا raw logs.
- لا credentials.
- لا tokens.
- لا shell حر.
- لا destructive actions.
- لا remediation في هذه المرحلة.
- لا direct AgentJob من Telegram.
- لا تشغيل أداة خارج ToolPolicy.
- لا عرض نتائج خام طويلة للمستخدم النهائي.
- كل نتيجة يجب أن تمر عبر تلخيص آمن قبل عرضها.

## 19. الترتيب العملي للأولوية القادمة

الأولوية التنفيذية المباشرة ليست Telegram، بل تجهيز المسار الداخلي أولا. الترتيب العملي المقترح:

1. تثبيت الوضع الحالي.
2. توثيق المعمارية المصححة.
3. بناء Safe Context Builder.
4. إنشاء Admin Chat داخلي.
5. ربط Tool Orchestrator.
6. اختبار تشغيل أداة موجودة من الشات.
7. إدخال Tool Builder داخل الشات.
8. تنفيذ أول دورة أداة كاملة.
9. إضافة التقارير من نفس الشات.
10. Pilot داخلي.
11. فتح Telegram لنفس AI.

## 20. الخلاصة النهائية

خطة تصحيح المسار لا تهدم أي شيء تم إنجازه. هي تعيد ترتيب المشروع حول فكرة واحدة: AI واحد داخل شات الأدمن يستخدم Tool Registry وRuntime الموجودين، يناقش الاحتياجات، ينشئ أدوات كـ command templates أو script templates آمنة، يختبرها، يشغلها، يفسر النتائج، وينشئ التقارير، ثم يتم فتح نفس العقل لاحقا من Telegram.

بهذا الشكل يكون المشروع قد حافظ على كل ما تم إنجازه، مع تحويله من Scanner يجمع بيانات إلى منتج تشخيص ذكي يمكن استخدامه داخليا ثم خارجيا بشكل آمن ومنظم.

## 21. مصطلحات ثابتة داخل الخطة

| المصطلح | المعنى المعتمد |
|---|---|
| Admin AI Chatbot | العقل الوحيد داخل النظام، ويعمل من خلال شات الأدمن/البورتال. |
| Safe Context | ملخص آمن من بيانات السيرفر يقرأه الـ AI بدون أسرار أو raw logs. |
| Tool Registry | قاعدة تعريفات الأدوات داخل المنصة. |
| Tool Runtime / Agent | المكون الموجود على السيرفر لتنفيذ command templates أو runtime handlers المعتمدة وإرجاع النتائج. |
| Tool Orchestrator | المسار الذي يربط طلب AI بالأدوات والسياسات والتنفيذ. |
| Tool Builder Workflow | مسار مناقشة وإنشاء واختبار واعتماد أدوات جديدة، أساسها الحالي command templates read-only. |
| Command Template | أمر read-only محفوظ ومنظم داخل Tool Registry مع مدخلات typed وقيود timeout/output/redaction. |
| Runtime Handler | collector أو parser متقدم داخل كود الـ Runtime، يستخدم لاحقا عند الحاجة وليس كأساس كل أداة. |
| Telegram Interface | واجهة خارجية لاحقة لنفس Admin AI Chatbot. |
